MINIMAL KANBAN 1.1
ПРАВИЛА РАБОТЫ CHATGPT С КОННЕКТОРОМ И MCP-СЦЕНАРИЯМИ

Назначение:
Этот файл нужен ChatGPT как практическая инструкция: как именно работать с коннектором, что считать правильным сценарием и как не ломать данные.

ГЛАВНЫЙ ПРИНЦИП
ChatGPT должен работать с этой доской аккуратно, но уверенно:
- сначала подтвердить контекст;
- потом читать;
- потом изменять;
- потом проверять результат.

СТАРТОВЫЙ WORKFLOW
В каждом новом чате:
1. ping_connector
2. bootstrap_context
3. если есть сомнение по runtime или tunnel — get_runtime_status
4. затем get_gpt_wall для входа в общий контекст

После этого можно переходить к рабочим операциям.

КАК СОЗДАВАТЬ КАРТОЧКУ АВТОМОБИЛЯ
1. Сначала извлечь максимум смысла из команды пользователя.
2. Заполнить базовые поля карточки:
   - vehicle
   - title
   - description
3. При необходимости добавить теги, срок и сигнал.
4. Затем использовать autofill_vehicle_data.
5. После автозаполнения обязательно прочитать карточку через get_card.

КАК РАБОТАЕТ АВТОЗАПОЛНЕНИЕ В 1.1
Главный режим:
card_content_first

Это значит:
autofill_vehicle_data сначала анализирует:
- vehicle
- title
- description
- raw_text, если он передан

И только потом формирует компактный профиль автомобиля.

Изображение или file-based сценарий больше не должен считаться основным пользовательским путём в карточке 1.1.
Если image-путь где-то ещё существует для совместимости, считать его вторичным fallback-сценарием.

КАКИЕ ПОЛЯ СЧИТАТЬ ОСНОВНЫМИ ДЛЯ АВТОМОБИЛЯ
Основной компактный набор:
- make_display
- model_display
- production_year
- vin
- engine_model
- gearbox_model
- drivetrain
- oem_notes

КАК РАБОТАТЬ С VIN
VIN может прийти:
- из команды пользователя;
- из description;
- из title;
- из уже сохранённой карточки.

Если VIN распознан неуверенно, не нужно выдумывать или silently исправлять его.
Лучше оставить предупреждение/низкую уверенность, чем вносить мусор.

КАК РАБОТАТЬ С РУЧНЫМИ ПОЛЯМИ
Если пользователь уже вручную исправил:
- VIN
- двигатель
- КПП
- привод
- заметку
то последующий autofill не должен silently затирать эти поля.

ПРАВИЛО:
manual data must remain authoritative unless user explicitly requests overwrite.

КАК РАБОТАТЬ С GET_CARD И GET_CARDS
ChatGPT должен учитывать, что карточка может вернуть:
- vehicle_profile
- vehicle_profile_compact

Для сценариев версии 1.1 основной ориентир — vehicle_profile_compact.
Полный vehicle_profile нужен в первую очередь для совместимости и расширенного внутреннего контракта.

КАК РАБОТАТЬ СО СТЕНОЙ
Перед сложной или широкой операцией сначала использовать get_gpt_wall.

СТЕНА помогает:
- понять ситуацию по всей доске;
- увидеть карточки и их контекст;
- понять историю изменений;
- определить, где находится нужная карточка;
- понять, какая машина и какие данные уже есть.

КАК ДЕЛАТЬ READ-AFTER-WRITE
После:
- create_card
- update_card
- autofill_vehicle_data
- move_card
- bulk_move_cards
- archive_card
- restore_card
желательно выполнить чтение результата:
- get_card для одной карточки;
- get_board_snapshot или get_gpt_wall для общего подтверждения.

КАКИЕ ОШИБКИ CHATGPT НЕ ДОЛЖЕН ДЕЛАТЬ
1. Не путать эту доску с другими канбан-коннекторами.
2. Не считать file upload основным сценарием автозаполнения карточки 1.1.
3. Не выдумывать VIN, модель двигателя, КПП или привод без признаков в данных.
4. Не игнорировать СТЕНУ как главный обзорный слой.
5. Не обновлять карточку без последующей проверки чтением, если менялись важные поля.

КАКОЙ РЕЗУЛЬТАТ СЧИТАТЬ ПРАВИЛЬНЫМ
Правильный рабочий цикл ChatGPT выглядит так:
1. подтвердить подключение;
2. войти в контекст доски;
3. найти нужную карточку или создать новую;
4. при необходимости запустить autofill_vehicle_data;
5. проверить compact vehicle profile;
6. при необходимости скорректировать данные;
7. снова подтвердить результат чтением.

КРАТКИЙ ЭТАЛОННЫЙ СЦЕНАРИЙ
ping_connector
-> bootstrap_context
-> get_gpt_wall
-> create_card или get_card
-> autofill_vehicle_data
-> get_card

Это и есть основной рабочий сценарий ChatGPT для версии 1.1.
